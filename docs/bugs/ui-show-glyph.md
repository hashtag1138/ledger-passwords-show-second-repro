# UI `show` Glyph Rendering Issue

## Summary

This issue is distinct from the `AZERTY` HID bug.

Observed on a real Ledger test device:

- master seed:
  `industry multiply science material way filter envelope firm destroy loop sauce poet olympic seven federal hidden mimic alley inject turn joke mask version first`
- nickname: `aaa`
- all charset options enabled

The device `show` screen visually displays a right-arrow glyph as part of the password, while the logical password generated by the app is:

```text
, )QUtmF+]25uI.)L3>3
```

So the logical substring is:

```text
+]
```

not a true Unicode arrow.

## What Is Proven

- `app-passwords` generates printable ASCII only.
- For the `aaa` case above, the logical password contains `+`.
- The visual arrow glyph seen on the real device is therefore not part of the password data model.

## Likely Layer

The anomaly appears in UI rendering rather than in password generation:

- generation is shared between `show` and `type password`
- the app passes the generated buffer directly to the UI layer
- the real device renders a glyph that does not match the ASCII character expected from the generated string

## Current Status

This repository documents the issue, but no patch is proposed yet.

Reason:

- the exact rendering mechanism has not been isolated
- it may be in NBGL / firmware rendering rather than in `app-passwords` application logic

## Reproduction Notes

The issue should be documented during real-device validation alongside the typed output.

Suggested checklist:

1. install a candidate build from this repo
2. create or load nickname `aaa`
3. open `Show password`
4. record the exact visual glyph sequence seen on the device
5. compare against the known logical string:
   `, )QUtmF+]25uI.)L3>3`
