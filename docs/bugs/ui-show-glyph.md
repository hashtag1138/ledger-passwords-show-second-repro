# UI `show` Glyph Rendering Issue

## Summary

This issue is distinct from the `AZERTY` HID bug.

Observed on a real Ledger test device:

- master seed:
  `industry multiply science material way filter envelope firm destroy loop sauce poet olympic seven federal hidden mimic alley inject turn joke mask version first`
- nickname: `aaa`
- all charset options enabled

At first glance, the device `show` screen looked like it was displaying a right-arrow glyph inside the password. After reviewing the device photo more carefully, this now looks much more like a visual confusion between adjacent ASCII characters than a true Unicode-like glyph.

The logical password generated by the app is:

```text
, )QUtmF+]25uI.)L3>3
```

So the relevant substring is:

```text
F+]
```

and not a true arrow character.

## Example Photo

The following photo shows the real-device screen that originally suggested an arrow-like glyph:

![Real-device example showing the `aaa` password screen](assets/ui-show-glyph-aaa-example.jpg)

In this example, the visually confusing area is the `F+]` group.

## What Is Proven

- `app-passwords` generates printable ASCII only.
- For the `aaa` case above, the logical password contains the ASCII substring `F+]`.
- So far, no true Unicode arrow or special display-only character has been proven in the password data model.

## Likely Cause of Confusion

At this stage, the most likely explanation is visual ambiguity rather than a distinct rendering bug.

Plausible causes include:

- the uppercase `F` and the following `+` are drawn very close to each other on the device screen;
- the vertical stroke of `F` plus the horizontal stroke of `+` can visually suggest an arrow-like shape when viewed quickly;
- the screen brightness, bloom, and photo softness make neighboring bright pixels bleed into each other;
- the low-resolution bitmap-like appearance of the font increases the chance of adjacent glyphs being mentally grouped into one symbol;
- the real UI already contains a navigation chevron on the right edge of the screen, which may make arrow-like interpretations feel more natural during quick inspection.

## Current Status

This repository keeps the note because the screen can still be misleading during manual validation, but the current interpretation is different from the original one.

Current conclusion:

- this is not currently treated as a proven separate UI bug in `app-passwords`;
- it is better described as a readability / visual confusion case around `F+]`;
- no application patch is proposed for this point alone.

## Reproduction Notes

This should still be documented during real-device validation alongside the typed output.

Suggested checklist:

1. install a candidate build from this repo
2. create or load nickname `aaa`
3. open `Show password`
4. record the exact visual appearance of the `F+]` area on the device
5. compare it against the known logical string:
   `, )QUtmF+]25uI.)L3>3`
6. avoid interpreting the `F+` cluster too quickly as a dedicated arrow glyph
